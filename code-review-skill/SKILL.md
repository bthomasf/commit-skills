# Code Review Skill — 代码审查

独立的代码审查能力，检测逻辑错误和无用代码，生成结构化报告并提供修复方案。

## 触发条件

当用户希望审查代码时使用此 Skill。典型触发场景：
- 用户说「审查代码」「帮我 review」「检查一下代码」
- 用户写完代码想检查质量
- 用户指定某个文件/目录要求审查

**不适用场景**：提交代码（用 commit-skill）、bug 定位（用 bug-trace-skill）。

## 工作流

```
[1. 确定审查范围] → [2. 过滤文件] → [3. 逻辑错误检测] → [4. 无用代码检测] → [5. 生成报告] → [6. 提供修复方案]
```

---

## 步骤 1：确定审查范围

支持两种模式：

### 默认模式（审查 git diff）

用户未指定范围时，审查当前所有变更：

1. 运行 `git rev-parse --is-inside-work-tree` 检查是否在 git 仓库中
   - 如果不是 git 仓库，提示「当前目录不是 git 仓库」，**终止流程**
2. 运行 `git diff --cached` 获取已暂存变更
3. 运行 `git diff` 获取未暂存变更
4. 合并为完整的变更文件列表
5. 如果两者均为空，提示「没有可审查的变更」，**终止流程**

### 指定范围模式

用户指定了文件或目录时：

1. 确认指定的文件/目录存在
2. 使用 Glob 工具列出范围内的所有代码文件
3. 使用 Read 工具读取这些文件内容

---

## 步骤 2：过滤文件

从审查范围中排除自动生成的文件：

- **lock 文件**: `package-lock.json`、`yarn.lock`、`pnpm-lock.yaml`、`Gemfile.lock`、`Pipfile.lock`、`poetry.lock`、`composer.lock`、`Cargo.lock`、`go.sum`
- **编译产物**: `dist/`、`build/`、`.next/`、`__pycache__/`、`.pyc`
- **依赖目录**: `node_modules/`、`vendor/`、`.venv/`
- **压缩/映射文件**: `.min.js`、`.min.css`、`.map`
- **二进制文件**: 图片、字体等

---

## 步骤 3：逻辑错误检测

阅读代码内容及上下文（使用 Read 工具读取完整文件），重点检查：

- **空值/空指针风险**：对可能为 null/undefined/None 的值直接调用方法或属性
- **边界条件遗漏**：数组越界、除零、空集合操作
- **类型不匹配**：字符串与数字比较、错误的类型断言
- **错误处理缺失**：try/catch 缺失、Promise 未处理 reject、错误被静默吞掉
- **条件判断错误**：逻辑运算符误用（&& vs ||）、条件永真/永假
- **资源未释放**：文件句柄、数据库连接、事件监听器未清理

发现的问题标记为**「问题」级别**。

---

## 步骤 4：无用代码检测

识别可能无用的代码，使用 Grep 工具在项目中搜索引用来验证：

- **未使用的导入**：import/require 语句中导入的符号在文件中未被引用
- **未使用的变量**：const/let/var/val 声明后未被引用
- **未使用的函数**：function/def/func 定义后在项目中未被调用

发现的问题标记为**「建议」级别**。

---

## 步骤 5：生成审查报告

将所有发现整理为结构化报告：

```
## 代码审查报告

### 问题（可能影响正确性）

1. **[文件路径:行号范围]** 问题描述
   > 相关代码片段

2. **[文件路径:行号范围]** 问题描述
   > 相关代码片段

### 建议（代码质量改进）

1. **[文件路径:行号范围]** 建议描述

---
共发现 N 个问题，M 个建议
```

如果未发现任何问题和建议，输出：「审查通过，未发现问题」。

---

## 步骤 6：提供修复方案

如果报告包含「问题」级别项，为每个问题提供具体的修复方案：

1. 展示问题代码片段
2. 展示修复后的代码片段
3. 解释修复原因

使用 AskQuestion 工具逐个询问用户是否应用修复：

| 选项 | 行为 |
|------|------|
| 应用修复 | 使用 StrReplace 工具执行代码修改 |
| 跳过 | 不做修改，继续处理下一个问题 |
| 全部跳过 | 结束修复流程 |
