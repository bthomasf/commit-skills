# Bug Trace Skill — Bug 追溯定位

根据用户描述的 bug 现象，主动搜索 `COMMIT-LOG.md` 定位可疑变更，分析代码根因并提供修复建议。

## 触发条件

当用户需要定位和修复 bug 时使用此 Skill。典型触发场景：
- 用户说「帮我查一下这个 bug」「定位一下问题」
- 用户描述了一个异常现象，希望找到原因
- 用户说「这个功能之前好好的，突然坏了」

**不适用场景**：提交代码（用 commit-skill）、代码审查（用 code-review-skill）。

## 前置条件

此 Skill 依赖 `COMMIT-LOG.md` 文件（由 commit-skill 维护）。如果文件不存在，将使用 `git log` 作为降级方案。

## 工作流

```
[1. 理解 Bug 描述] → [2. 搜索 COMMIT-LOG] → [3. 展示可疑记录] → [4. 代码分析] → [5. 输出修复建议]
```

---

## 步骤 1：理解 Bug 描述

从用户的描述中提取关键信息：

- **功能模块**：涉及哪个功能区域（登录、支付、搜索等）
- **文件/路径线索**：用户提到的具体文件或路径
- **错误现象**：白屏、报错、数据丢失、行为异常等
- **操作路径**：触发 bug 的用户操作步骤
- **时间线索**：什么时候开始出现的（有助于缩小 commit 范围）

**如果描述过于模糊**，使用 AskQuestion 工具追问：

| 信息不足 | 追问示例 |
|----------|----------|
| 不知道哪个页面 | 「这个问题出现在哪个页面或功能上？」 |
| 不知道操作步骤 | 「能描述一下触发 bug 的操作步骤吗？」 |
| 不知道何时开始 | 「这个问题是最近才出现的，还是一直有？」 |

从以上信息中提取搜索关键词（中文 + 英文，如「登录」+「login」+「auth」）。

---

## 步骤 2：搜索 COMMIT-LOG

### 主方案：搜索 COMMIT-LOG.md

1. 使用 Read 工具读取项目根目录的 `COMMIT-LOG.md`
2. 使用 Grep 工具按关键词搜索，覆盖记录中的所有字段：
   - 改动摘要
   - 影响范围（文件路径和变更说明）
   - 行为变化
3. 收集所有匹配的 commit 记录

### 降级方案：COMMIT-LOG.md 不存在

如果文件不存在，使用 git 命令直接搜索：

```bash
git log --all --oneline --grep="<关键词>"
git log --all --oneline -- <相关文件路径>
```

---

## 步骤 3：展示可疑记录

将匹配的 commit 记录按相关度排序后展示给用户：

```
## 可疑变更记录

### 1. feat(auth): 添加 token 刷新机制 (abc1234) — zhangsan@co.com
日期: 2026-03-01
改动摘要: 当 access token 过期时自动刷新...
行为变化: API 请求 401 时不再直接跳转登录页...
相关度: ★★★ (匹配关键词: 登录, token, 401)

### 2. fix(api): 修复请求超时处理 (def5678) — lisi@co.com
日期: 2026-02-28
...
相关度: ★★ (匹配关键词: 请求, api)
```

如果匹配记录较多，优先展示最近的、相关度最高的。

---

## 步骤 4：代码分析与定位

对每个可疑 commit 进行深入分析：

1. 运行 `git show <hash>` 查看完整 diff
2. 使用 Read 工具读取相关源文件的当前版本，理解完整上下文
3. 结合 bug 描述分析：
   - 这个 commit 改了什么逻辑？
   - 改动是否可能导致用户描述的现象？
   - 是单个 commit 的问题还是多个 commit 累积的影响？

如果单个 commit 不够定位，分析多个相关 commit 的组合影响。

---

## 步骤 5：输出修复建议

生成结构化的分析报告：

```
## Bug 分析报告

### 问题定位
- **根因 commit**: <short-hash> <type>(<scope>): <subject>
- **作者**: <email>
- **日期**: YYYY-MM-DD
- **问题文件**: <文件路径>:<行号范围>

### 原因分析
<详细解释为什么这段代码会导致 bug，引用具体的代码片段>

### 修复方案

**方案描述**: <修复思路>

**修改前**:
> <问题代码>

**修改后**:
> <修复代码>

### 影响评估
- <修复可能影响的其他功能>
```

使用 AskQuestion 工具询问用户：

| 选项 | 行为 |
|------|------|
| 应用修复 | 使用 StrReplace 工具执行代码修改 |
| 我自己修 | 结束流程，用户手动处理 |
| 继续分析 | 分析下一个可疑 commit |
